# shortcuts/

One folder per shortcut SET, named after the set (e.g. `Rainbow/`,
`DUGA/`), each holding one VBS per key the user picked:

```
📁 shortcuts/
  📁 Rainbow/
    🔧 F1.vbs … F12.vbs     ← runs the Resolver with --shortcut "Rainbow:F1"
```

The user's flow (see [Shortcuts Tab](../gui/shortcuts_tab.py)): name the
set → pick selector → pick ANY keys → pick a color per key → "Create
shortcut files". Normal selectors are then registered automatically by
the [Hotkey Daemon](../hotkey_daemon.md); for **hypershift** sets the GUI
opens this folder plus Razer Synapse with a linking guide — each key's
file is linked in Synapse ONCE, and re-mapping colors later is a pure
config change (the files never need re-linking).

A slot whose set or key was removed is a quiet no-op. Folders are
regenerated by the GUI or `python resolver.py --write-slots`; folders of
deleted sets are cleaned up. Generated files get no individual docs
(project convention); the folders are gitignored (they embed the local
Python path).
